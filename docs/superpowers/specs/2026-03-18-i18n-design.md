# i18n Design — CPS Admin Dashboard

## Overview

Add internationalization to the CPS Admin Dashboard (React 19 + Ant Design v6 + FastAPI).

- **Languages**: zh-CN (default), en-US, es-ES
- **Library**: react-i18next (i18next + react-i18next)
- **Scope**: ~180 hardcoded strings across 6 pages + 5 components, plus Ant Design built-in text

## Architecture

### Frontend

```
web/src/i18n/
├── index.ts              # i18n initialization
├── i18n.d.ts             # TypeScript type declarations for translation keys
├── useLocale.ts          # Custom hook: manages i18next + Ant Design locale atomically
└── locales/
    ├── zh-CN.json        # Chinese (default)
    ├── en-US.json        # English
    └── es-ES.json        # Spanish
```

### Backend

- `admin_users.locale` column: `VARCHAR(10) NOT NULL DEFAULT 'zh-CN'`
- DB constraint: `CHECK (locale IN ('zh-CN', 'en-US', 'es-ES'))`
- New endpoint: `PATCH /api/v1/auth/locale`
- Alembic migration 005

### Storage & Sync

```
Priority chain (highest to lowest):
1. User action (login page dropdown or admin topbar dropdown)
2. localStorage cache
3. Server DB (admin_users.locale)
4. Default: 'zh-CN'

Sync direction: LOCAL-FIRST
- Login page: read localStorage → render → user may switch → write localStorage
- Login success:
    → if localStorage has value: PUT local value to server (local wins)
    → if localStorage is empty: use server value, write to localStorage
- Admin pages: read from useLocale hook (backed by localStorage)
- Language switch (logged in): update localStorage + PUT to server + i18n.changeLanguage + ConfigProvider
- Language switch (not logged in): update localStorage + i18n.changeLanguage + ConfigProvider (no PUT)
```

### Login-to-Locale Sync Flow

```
Login page loads
  → getSafeLocale(localStorage) → i18n.init with that locale
  → User may switch language via LangSwitcher → writes localStorage

User clicks "Login"
  → POST /auth/login → success → response includes user.locale
  → useAuth.login() calls useLocale.syncAfterLogin(serverLocale):
      raw = localStorage.getItem('locale')
      if raw !== null → PATCH /auth/locale with getSafeLocale(raw) (local wins)
      if raw === null → changeLocale(getSafeLocale(serverLocale)) (server wins, writes localStorage)
  → Navigate to dashboard
```

## Component Design

### i18n Initialization (index.ts)

```typescript
// web/src/i18n/index.ts
import i18n from 'i18next'
import { initReactI18next } from 'react-i18next'
import zhCN from './locales/zh-CN.json'
import enUS from './locales/en-US.json'
import esES from './locales/es-ES.json'
import { getSafeLocale } from './useLocale'

const stored = typeof localStorage !== 'undefined' ? localStorage.getItem('locale') : null

i18n.use(initReactI18next).init({
  resources: {
    'zh-CN': { translation: zhCN },
    'en-US': { translation: enUS },
    'es-ES': { translation: esES },
  },
  lng: getSafeLocale(stored),
  fallbackLng: 'zh-CN',
  interpolation: { escapeValue: true },  // keep XSS protection
  returnNull: false,
})

export default i18n
```

### useLocale Hook

Central hook that atomically manages i18next language and Ant Design locale.

```typescript
// web/src/i18n/useLocale.ts
export const SUPPORTED_LOCALES = ['zh-CN', 'en-US', 'es-ES'] as const
export type SupportedLocale = typeof SUPPORTED_LOCALES[number]

// Whitelist validation — used in init and changeLocale
export function getSafeLocale(raw: string | null): SupportedLocale {
  return SUPPORTED_LOCALES.includes(raw as SupportedLocale)
    ? (raw as SupportedLocale)
    : 'zh-CN'
}

interface UseLocaleReturn {
  locale: SupportedLocale
  antdLocale: Locale             // Ant Design locale object for ConfigProvider
  changeLocale: (l: SupportedLocale) => Promise<void>
  syncAfterLogin: (serverLocale: string) => Promise<void>
}

// changeLocale does ALL of:
// 1. i18n.changeLanguage(locale)
// 2. Update Ant Design locale state (triggers ConfigProvider re-render)
// 3. Write to localStorage
// 4. If user is logged in (check via auth context user !== null, NOT via cookie —
//    cps_session cookie is httponly and invisible to JS), PATCH /auth/locale
//    (fire-and-forget, catch errors silently)
//
// Implementation: useLocale accepts isLoggedIn param or reads from shared auth context.

// syncAfterLogin(serverLocale):
// Called by useAuth.login() after successful login.
// IMPORTANT: check raw localStorage value, not getSafeLocale() result:
//   const raw = localStorage.getItem('locale')
//   if (raw !== null) → PATCH server with getSafeLocale(raw) (local wins)
//   if (raw === null) → changeLocale(getSafeLocale(serverLocale)) (server wins)
```

### ECharts i18n Integration

ECharts does not use react-i18next directly. Translation values must be injected when building the `option` object inside the component function body (where hooks are available):

```typescript
// PriceChart.tsx — inside the component
const { t } = useTranslation()
const option = {
  legend: { data: [t('chart.amazon'), t('chart.new'), t('chart.used')] },
  series: data.map(s => ({ ...s, name: t(`chart.${s.type}`) })),
  // Replace existing string formatter '$${value}' with function for locale-aware formatting
  // v is in dollars (data already divided by 100 in PriceChart), multiply back for formatPrice(cents)
  yAxis: { axisLabel: { formatter: (v: number) => formatPrice(v * 100, i18n.language) } },
}

// Dashboard.tsx throughput chart
series: [{ name: t('dashboard.seriesCompleted'), ... }]
```

### PATCH /api/v1/auth/locale

```python
class LocaleUpdateRequest(BaseModel):
    locale: Literal["zh-CN", "en-US", "es-ES"]

@router.patch("/locale")
async def update_locale(
    body: LocaleUpdateRequest,
    request: Request,
    db: Annotated[AsyncSession, Depends(get_db)],
    current_user: Annotated[AdminUser, Depends(get_current_user)],
):
    current_user.locale = body.locale
    await db.commit()
    client_ip = request.client.host if request.client else "unknown"
    await log_audit(db, current_user.id, "update_locale", "user", current_user.id, client_ip)
    return {"locale": current_user.locale}
```

## Security Checklist

- [x] `LocaleUpdateRequest` uses `Literal["zh-CN", "en-US", "es-ES"]` — Pydantic rejects invalid values
- [x] DB CHECK CONSTRAINT limits column values
- [x] PATCH endpoint uses `Depends(get_current_user)` — requires auth
- [x] Endpoint NOT added to `_CSRF_EXEMPT` — CSRF protection active
- [x] Frontend `getSafeLocale()` validates localStorage value against whitelist
- [x] i18n init does NOT set `interpolation.escapeValue = false`
- [x] No unsafe HTML injection with translation strings
- [x] Locale change logged to audit trail

## Out of Scope

- RTL layout support
- Translation management UI
- Lazy-loading translation files (not needed for 3 languages, ~15KB total)
- Backend API response i18n (API stays English, frontend translates display values)
